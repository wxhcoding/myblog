# SpringBoot

# 快速入门

  1. 创建Maven工程
  2. 添加依赖(springboot父工程依赖 , web启动器依赖)
<!-- 所有springboot项目都必须继承自 spring-boot-starter-parent 
    通过这个parent管理插件的信息 再通过这个parent的parent管理我们的依赖版本
-->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.0.5</version>
</parent>

<dependencies>
<!--        web开发的场景启动器 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
  1. 编写启动引导类(springboot项目运行的入口)

该类应放在和controller包同一级目录下，因为该启动类会默认扫描所在目录以及目录下面的所有包的注解

@SpringBootApplication //springboot启动器必须加此注解
public class Application {
    public static void main(String[] args) {
        // 调用SpringApplication的run方法  分别传入当前类 和main方法中的args参数
        SpringApplication.run(Application.class, args);
    }
}
  1. 编写处理器Controller
@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello(){
        return "Hello,Spring Boot 3!";
    }
}
  1. 启动项目

# 配置文件

Springboot支持两种类型的配置文件(并且必须命名为下面的形式且放在resources目录下)

  • application.properties属性配置文件
  • application.yml配置文件(application.yaml配置文件)

# properties文件

# 下面这个配置是springboot默认的 可以直接读取并修改配置信息
server.port=8081

# 下面的是自定义的配置信息 springboot不会读取到 我们需要自己在一个配置类中去读取
spring.jdbc.datasource.driverClassName=com.mysql.cj.jdbc.driver
spring.jdbc.datasource.url=jdbc:mysql:///springboot_01
spring.jdbc.datasource.username=root
spring.jdbc.datasource.password=root

@Component // 加入到IOC容器
@Data // 使用lombok 添加get/set方法
public class PropertiesConfig {
    // 注意 value的注解 导包一定要注意不要导错
    // org.springframework.beans.factory.annotation.Value
    @Value("${spring.jdbc.datasource.driverClassName}")
    private String driverClassName;

    @Value("${spring.jdbc.datasource.url}")
    private String url;

    @Value("${spring.jdbc.datasource.username}")
    private String username;

    @Value("${spring.jdbc.datasource.password}")
    private String password;
}

提示

下面可以进行配置文件的批量注入

@Component
@Data
// 批量注入配置文件 并设置前缀
@ConfigurationProperties(prefix = "spring.jdbc.datasource")
public class MultiPropertiesConfig {
    // 如果我们的属性名和 配置信息中的命名 除去前缀 一样的话 可以不用设置@value注解
    private String driverClassName;
    private String url;
    private String username;
    private String password;
}

测试可以获取到对应的信息

@RestController
public class HelloController {

    // 引入配置的对象
    @Autowired
    private PropertiesConfig propertiesConfig;

    // 批量引入配置文件信息
    @Autowired
    private MultiPropertiesConfig multiPropertiesConfig;

    @RequestMapping("pro")
    public String getProperties(){
        // 测试输出配置文件的内容
        System.out.println(propertiesConfig);
        return "hello SpringBoot properties";
    }

    @RequestMapping("multi")
    public String getMultiProperties(){
        // 测试输出配置文件的内容
        System.out.println(multiPropertiesConfig);
        return "hello SpringBoot properties2";
    }
}

# 自动配置

# 初步理解

  • 在之前我们创建web项目的时候需要引用DispatcherServlet、JackJson等相关的组件，而我们在使用SpringBoot创建项目的时候并没有引入这些，那是因为SpringBoot帮我们进行了自动配置，我们在创建web项目时一定需要引入的一个依赖就是spring-boot-starter-web，而这个依赖里面已经帮我们自动引入了这些依赖，而SpringBoot识别到了这些的导入会自动帮我们进行配置

  • 使用SpringBoot时我们需要创建一个引导启动类，而这个类上我们需要添加一个注解@SpringBootApplication，此时我们点进去这个类可以发现是由三个注解组成的，@SpringBootConfiguration、@EnableAutoConfiguration、@ComponentScan

    @SpringBootApplication
public class MainApplication {
    public static void main(String[] args) {
        SpringApplication.run(MainApplication.class,args);
    }
}

        @ConditionalOnClass(DispatcherServlet.class)
public class DispatcherServletAutoConfiguration {}
         

      • 比如这个DispatcherServletAutoConfiguration，我们会发现上面有一个注解@ConditionalOnClass，说明我们所需的配置是按需进行加载的，只有我们导入了这个配置类，SpringBoot识别到这个类DispatcherServlet.class加载了，就会帮我们自动导入

# 自动配置流程

  • 下面以spring-boot-starter-web为例进行介绍

启动器

  1. 导入spring-boot-starter-web,会自动帮我们导入web开发所需要的一些依赖

    • 会自动导入需要的这些依赖，如json、tomcat、webmvc等

    • 每个场景启动器都引入了一个spring-boot-starter，核心场景启动器

    • 这个核心启动器会导入spring-boot-autoconfigure这个包

    • 这个包里面是各种场景的AutoConfiguration自动配置

    • 而SpringBoot却默认不会扫描到这个spring-boot-autoconfigure下的配置类，因为默认扫描的包是启动类所在的包，这就需要我们启动类上的另外一个注解@EnableAutoConfiguration发挥作用

主程序相关注解

  1. 主程序@SpringBootApplication

    • @SpringBootApplication由三个注解组成@SpringBootConfiguration、@EnableAutoConfiguration、@ComponentScan,上面简单介绍了它们三个的不同作用

    • 默认扫描使用注解@ComponentScan，扫描主程序所在的包以及子包，而不会扫描到上面提到的spring-boot-autoconfigure这个包的配置类

    • @EnableAutoConfiguration：SpringBoot 开启自动配置的核心

      • 通过这个注解下面的@Import({AutoConfigurationImportSelector.class})导入的这个类

      • AutoConfigurationImportSelector.class这个类会默认去这个META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports文件里面去找对应的配置类，而我们导入的这个文件里面有指定的142个自动配置类

      • 项目启动的时候利用@Import批量导入组件机制把spring-boot-autoconfigure这个包下面的142个xxxAutoConfiguration类导入进来

      • 虽然有142个自动配置类，但他们都是按需生效，因为在每个xxxAutoConfiguration类上都有一个注解@ConditionalOnClass，只有条件成立才会生效

自动配置类

  1. 自动配置类xxxAutoConfiguration

    • 这个类下面会使用@Bean注解，往容器中放入一堆组件

    • 而几乎每个配置类上都有一个注解@EnableConfigurationProperties(xxxProperties.class),这个的作用是我们的组件是从这个xxxProperties.class类中去提取需要的属性值，即我们组件需要的核心参数都是从xxxProperties.class中去获取

    • 而这个xxxProperties.class又是和我们的配置文件application.properties绑定的

    • 只要我们在配置文件中修改了某些参数，那么这些核心组件的底层的参数就可以修改了

综上所述，我们在写业务时只需要写业务，如果需要修改配置某些参数只需要在application.prperoties中修改配置，而SpringBoot会自动帮我们把其他流程整合好，并且使我们配置的参数生效

核心流程

  1. 导入启动器starter，就会导入autoconfigure包
  2. autoconfigure 包里面 有一个文件 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports,里面指定的所有启动要加载的自动配置类
  3. 主程序上的@EnableAutoConfiguration 会自动的把上面文件里面写的所有自动配置类都导入进来
  4. 但不会全部导入，会根据这个类xxxAutoConfiguration上的@Conditionalxxx注解有条件注解进行按需加载
  5. xxxAutoConfiguration给容器中导入一堆组件，组件都是从 xxxProperties中提取属性值
  6. xxxProperties又是和配置文件进行了绑定

最终效果: 导入starter、修改配置文件，就能修改底层行为，并自动导入

# 静态资源目录

在WEB开发中我们需要引入一些静态资源 , 例如 : HTML , CSS , JS , 图片等 , 如果是普通的项目静态资源可以放在项目的webapp目录下。现在使用SpringBoot做开发 , 项目中没有webapp目录 , 我们的项目是一个jar工程，那么就没有webapp，我们的静态资源应该放在SpringBoot的默认查找目录
默认的静态资源路径为(下面的classpath是针对的是编译之后target目录下的classes文件夹)：

  • · classpath:/META-INF/resources/
  • · classpath:/resources/
  • · classpath:/static/
  • · classpath:/public/

我们只要静态资源放在这些目录中任何一个，SpringMVC都会帮我们处理。 我们习惯会把静态资源放在classpath:/static/ 目录下。在resources目录下创建index.html文件
浏览器直接访问 http://localhost:8080/index.html 即可

如果想要修改默认的查找目录，可以在配置文件中进行修改：

spring:
  web:
    resources:
      static-locations: classpath:/webapp/

# 拦截器

因为springboot没有xml文件进行配置，所以我们需要使用配置类的方式来实现拦截器

  • 编写一个拦截器
    @Component
public class MyInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        System.out.println("MyInterceptor.preHandle");
        return true;
    }

    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        System.out.println("MyInterceptor.postHandle");
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        System.out.println("MyInterceptor.afterCompletion");
    }
}
  • 通过WebMvcConfigurer注册拦截器
    @Configuration
public class MvcConfig implements WebMvcConfigurer {

    @Autowired
    private MyInterceptor myInterceptor;

    /**
     * 拦截当前目录及子目录下的所有路径 /user/**   /user/findAll  /user/order/findAll
     * 拦截当前目录下的以及子路径   /user/*     /user/findAll
     */

    // 通过重写其中的addInterceptors来实现拦截器的注册
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(myInterceptor).addPathPatterns("/user");
    }
}

浏览器访问 http://localhost:8080/user 查看控制台输出

package com.atcode.blog.utils;

import lombok.Getter;

/**
 * 统一返回结果状态信息类
 */
@Getter
public enum ResultCodeEnum {

    SUCCESS(200,"成功"),
    FAIL(201, "失败"),
    ;

    private Integer code;
    private String message;
    private ResultCodeEnum(Integer code, String message) {
        this.code = code;
        this.message = message;
    }
}

package com.atcode.blog.utils;

import io.swagger.v3.oas.annotations.media.Schema;
import lombok.Data;

/**
 * 全局统一返回结果类
 */
@Data
@Schema(description = "统一的返回数据类型")
public class Result<T> {

    @Schema(description = "状态码")
    private Integer code;

    @Schema(description = "返回信息")
    private String message;

    @Schema(description = "返回的数据")
    private T data;

    public Result(){}

    // 返回数据
    protected static <T> Result<T> build(T data) {
        Result<T> result = new Result<T>();
        if (data != null)
            result.setData(data);
        return result;
    }

    public static <T> Result<T> build(T body, Integer code, String message) {
        Result<T> result = build(body);
        result.setCode(code);
        result.setMessage(message);
        return result;
    }

    public static <T> Result<T> build(T body, ResultCodeEnum resultCodeEnum) {
        Result<T> result = build(body);
        result.setCode(resultCodeEnum.getCode());
        result.setMessage(resultCodeEnum.getMessage());
        return result;
    }

    public static<T> Result<T> ok(){
        return Result.ok(null);
    }

    /**
     * 操作成功
     */
    public static<T> Result<T> ok(T data){
        Result<T> result = build(data);
        return build(data, ResultCodeEnum.SUCCESS);
    }

    public static<T> Result<T> fail(){
        return Result.fail(null);
    }

    /**
     * 操作失败
     */
    public static<T> Result<T> fail(T data){
        Result<T> result = build(data);
        return build(data, ResultCodeEnum.FAIL);
    }

    public Result<T> message(String msg){
        this.setMessage(msg);
        return this;
    }

    public Result<T> code(Integer code){
        this.setCode(code);
        return this;
    }

    public boolean isOk() {
        if(this.getCode().intValue() == ResultCodeEnum.SUCCESS.getCode().intValue()) {
            return true;
        }
        return false;
    }
}

# 整合Swagger

Swagger是一种基于OpenAPI规范的API文档生成工具，它可以根据Java代码中的注解自动生成API接口文档，并提供UI界面进行在线测试和调试。Swagger为开发人员提供了更加方便、直观的API管理方式，有助于提升API的可读性和可维护性。
Knife4j是一种基于Swagger构建的增强工具，它在Swagger的基础上增加了更多的功能和扩展，提供了更加丰富的API文档管理功能。相比于原版Swagger，Knife4j的主要特点包括界面更加美观，支持多种注解，提供更多的插件扩展，集成了spring-boot-starter

依赖导入

首先我们在项目中导入knife4j的依赖

<dependency>
    <groupId>com.github.xiaoymin</groupId>

    <version>4.1.0</version>
</dependency>

配置

添加knife4j的配置类

package com.atcode.blog.config;

import io.swagger.v3.oas.models.OpenAPI;
import io.swagger.v3.oas.models.info.Contact;
import io.swagger.v3.oas.models.info.Info;
import org.springdoc.core.models.GroupedOpenApi;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class Knife4jConfig {

    @Bean
    public GroupedOpenApi adminApi() {      // 创建了一个api接口的分组
        return GroupedOpenApi.builder()
                .group("blog-api")         // 分组名称
                .pathsToMatch("/**")  // 接口请求路径规则
                .build();
    }

    /***
     * @description 自定义接口信息
     */
    @Bean
    public OpenAPI customOpenAPI() {

        return new OpenAPI()
                .info(new Info()
                        .title("微博客API接口文档")
                        .version("1.0")
                        .description("微博客API接口文档")
                        .contact(new Contact().name("JhWang"))); // 设定作者
    }

}

常用注解

  • @Tag(name="")： 用在controller类上，对controller进行说明
  • @Operation(summary=""): 用在controller接口方法上对接口进行描述
  • @Parameters(value={})：用在controller接口方法上对单个参数进行描述
  • @Schema(description="")： 用在实体类和实体类属性上，对实体类以及实体类属性进行描述
