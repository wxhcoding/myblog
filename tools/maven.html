# Maven

# 主要作用

  1. 依赖管理： Maven 可以管理项目的依赖，包括自动下载所需依赖库、自动下载依赖需要的依赖并且保证版本没有冲突、依赖版本管理等。通过 Maven，我们可以方便地维护项目所依赖的外部库，而我们仅仅需要编写配置即可。
  2. 构建管理： 项目构建是指将源代码、配置文件、资源文件等转化为能够运行或部署的应用程序或库的过程！
    Maven 可以管理项目的编译、测试、打包、部署等构建过程。通过实现标准的构建生命周期，Maven 可以确保每一个构建过程都遵循同样的规则和最佳实践。同时，Maven 的插件机制也使得开发者可以对构建过程进行扩展和定制。主动触发构建，只需要简单的命令操作即可。

# 使用配置

首先把下载好的压缩包，解压到自己电脑上的一个文件夹内，然后开始配置环境变量(注意：这里首先要有java的环境，即JAVA_HOME的环境变量)。
在系统变量中配置一个变量为MAVEN_HOME, 值为自己解压所在的文件夹，例如D:\maven\apache-maven-3.6.3, 这个配置完成后，在Path的环境变量中添加一条%MAVEN_HOME%\bin来配置maven，然后测试自己的配置，显示如下就说明配置成功。

# 修改配置文件

我们需要需改maven/conf/settings.xml配置文件，来修改maven的一些默认配置。我们主要休要修改的有三个配置：
1.依赖本地缓存位置（本地仓库位置）
2.maven下载镜像
3.maven选用编译项目的jdk版本！

<!-- localRepository
   | The path to the local repository maven will use to store artifacts.
   |
   | Default: ${user.home}/.m2/repository
  <localRepository>/path/to/local/repo</localRepository>
  -->
<!--  本地仓库地址 -->
<localRepository>D:\maven\repository</localRepository>


<!--在mirrors节点(标签)下添加中央仓库镜像-->
<mirror>
    <id>alimaven</id>
    <name>aliyun maven</name>
    <url>http://maven.aliyun.com/nexus/content/groups/public/</url>
    <mirrorOf>central</mirrorOf>
</mirror>


<!--在profiles节点(标签)下添加jdk编译版本-->
<profile>
    <id>jdk-17</id>
    <activation>
        <activeByDefault>true</activeByDefault>
        <jdk>17</jdk>
    </activation>
    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <maven.compiler.compilerVersion>17</maven.compiler.compilerVersion>
    </properties>
</profile>

# GAVP属性

Maven 中的 GAVP 是指 GroupId、ArtifactId、Version、Packaging 等四个属性的缩写，其中前三个是必要的，而 Packaging 属性为可选项。
这四个属性主要为每个项目在maven仓库总做一个标识，类似人的姓-名, 有了具体标识，方便后期项目之间相互引用依赖等

GAV遵循一下规则：

  1. GroupID 格式：com.{公司/BU }.业务线.[子业务线]，最多 4 级。

    • 说明：{公司/BU} 例如：alibaba/taobao/tmall/aliexpress 等 BU 一级；子业务线可选。
    • 正例：com.taobao.tddl 或 com.alibaba.sourcing.multilang

  2. ArtifactID 格式：产品线名-模块名。语义不重复不遗漏，先到仓库中心去查证一下。

    • 正例：tc-client / uic-api / tair-tool / bookstore / xxx

  3. Version版本号格式推荐：主版本号.次版本号.修订号

    • 主版本号：当做了不兼容的 API 修改，或者增加了能改变产品方向的新功能。
    • 次版本号：当做了向下兼容的功能性新增（新增类、接口等）。
    • 修订号：修复 bug，没有修改方法签名的功能加强，保持 API 兼容性。
      例如： 初始→1.0.0 修改bug → 1.0.1 功能调整 → 1.1.1等

Packaging定义规则： 指示将项目打包为什么类型的文件，idea根据packaging值，识别maven项目类型

  • packaging 属性为 jar（默认值），代表普通的Java工程，打包以后是.jar结尾的文件。
  • packaging 属性为 war，代表Java的web工程，打包以后.war结尾的文件。
  • packaging 属性为 pom，代表不会打包，用来做继承的父工程。

# 构建Java EE工程

# 手动创建

首先创建一个maven的工程，然后自己手动创建添加web的项目结构文件，在src/main的下面创建webapp文件夹，在webapp文件夹下面创建WEB-INF文件，里面放web.xml文件。
然后修改pom.xml文件，更改打包的方式

<modelVersion>4.0.0</modelVersion>  
<groupId>com.atcode</groupId>  
<artifactId>headline-maven</artifactId>  
<version>1.0-SNAPSHOT</version>  
<!--新增下面war的打包方式-->
<packaging>war</packaging>

刷新maven工程，在webapp文件夹上出现小红点说明创建成功。

# 插件创建

在idea中的plugins中搜索JBLJavaToWeb插件，就可以右击自己创建的项目，点击JBLJavaToWeb，即可自动帮我们创建成功。

# pom.xml

<?xml version="1.0" encoding="UTF-8"?>

<project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">

    <!--maven配置文件的版本 固定4.0.0-->
    <modelVersion>4.0.0</modelVersion>

    <!--maven gavp-->
    <!--一般情况下不会修改-->
    <groupId>com.atcode</groupId>
    <artifactId>maven-project-javaee-01</artifactId>
    <!--时机： 部署的时候进行修改  install deploy-->
    <!--推荐：三层命名 主版本.次版本.修订号-->
    <version>1.0-SNAPSHOT</version>
    <!--默认打包方式javaSE jar - web项目的打包方式javaEE war - 不打包 pom -->
    <packaging>war</packaging>

    <!--声明配置参数-->
    <properties>
        <!--标签名是变量的名 变量值 标签名是变量的名-->
        <mysql.version>8.0.30</mysql.version>
        <spring.varsion>6.0.6</spring.varsion>

        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <!--添加依赖信息标签
        导入依赖 导入其他的maven工程 填写其他工程的gav属性即可
        gav如何知道
        1. maven提供的官方网站 https://mvnrepository.com/ [极其不稳定]
        2. 插件 plugins-> maven-search  在导航栏上方的tools下面找到该插件 使用copy of maven
        3. 记在脑子里

        提升：
            1.统一的配置声明
              声明配置  <properties> <框架名.version>版本号</框架名.version>
              引用版本 <version>${框架名.version}</version>
            2.引用的作用范围 - 拿不准的情况下就不适用这个scope
               <scope> jar包的范围 </scope>
                  jar包的作用范围 主程序[main] 测试程序[test] 打包运行[jar war]
                  compile[默认值]  主程序 测试 打包运行  【大多数】
                  provided        主程序 测试 不要     【servlet】
                  runtime         不要  不要  打包运行 【mysql】
                  test            不要  测试  不要    【junit】

    -->
    <dependencies>
        <!--依赖的集合-->
        <!-- 一个具体的依赖 一个具体的jar 一个其他maven工程的gav-->
        <dependency>
            <groupId>mysql</groupId>
            <artifactId>mysql-connector-java</artifactId>
            <!--引用上面声明的版本-->
            <version>${mysql.version}</version>
            <!--声明生效范围-->
            <scope>runtime</scope>
        </dependency>

        <dependency>
            <groupId>junit</groupId>
            <artifactId>junit</artifactId>
            <version>4.12</version>
          <!--声明生效范围-->
            <scope>test</scope>
        </dependency>

        <!-- jackson [core annotations databind] -->
        <!-- 依赖传递  只需导入 databind 就可自动导入其他两个依赖-->

    </dependencies>

    <!--
      构建配置 build
        1.打包命名 finalName
          自动化部署 【要求打包的压缩包的命名是固定的！】
        2.指定打包资源
          maven项目结构
            src
              main
                java
                resources
                webapp
              test
            pom.xml
            target
          默认情况下 按照文件夹放置对应的文件
    -->

    <build>
        <!--默认规则 artifactId-version.packaging
          这里可以自定义名称
        -->
        <finalName>testName</finalName>

        <!--自己编写资源打包的规则-->
        <resources>
            <!--一套具体的规则-->
            <resource>
                <!--  src/java | resources | webapp -->
                <directory>src/main/java</directory>
                <includes>
                    <!--包含文件的规则
                        包名/文件名.后缀名

                        **/*.*
                        ** 多层任意包  * 任意文件名  .* 任意后缀
                    -->
                    <include>**/*.*</include>
                </includes>
                <excludes>
                    <!--去除源文件 只要编译之后的 .class 文件-->
                    <exclude>**/*.java</exclude>
                </excludes>
            </resource>

            <resource>
                <directory>src/main/resources</directory>
                <includes>
                    <include>**/*.*</include>
                </includes>
            </resource>
        </resources>

        <!--在plugins标签下配置插件的信息-->
        <!-- jdk17 和 war包版本插件不匹配 -->
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-war-plugin</artifactId>
                <version>3.2.2</version>
            </plugin>
        </plugins>
    </build>
</project>

# Maven工程构建

项目构建是指将源代码、依赖库和资源文件等转换成可执行或可部署的应用程序的过程，在这个过程中包括编译源代码、链接依赖库、打包和部署等多个步骤。

命令描述
mvn clean清理编译或打包后的项目结构
mvn compile编译项目，生成target文件
mvn test执行测试源码 (测试)
mvn package打包项目，生成war文件
mvn install打包后上传到maven本地仓库(本地部署)
mvn deploy只打包，上传到maven私服仓库(私服部署)
mvn site生成站点 (报告)

构建周期

主要三个构建生命周期：

  • 清理周期：主要是对项目编译生成文件进行清理
    • 包含命令：clean
  • 默认周期：定义了真正构件时所需要执行的所有步骤，它是生命周期中最核心的部分
    • 包含命令：compile - test - package - install / deploy
  • 报告周期
    • 包含命令：site 打包: mvn clean package 本地仓库: mvn clean install

# 继承和聚合

继承

Maven 继承是指在 Maven 的项目中，让一个项目从另一个项目中继承配置信息的机制。继承可以让我们在多个项目中共享同一配置信息，简化项目的管理和维护工作。
在父工程中统一管理项目中的依赖信息，父工程通过dependencyManagement标签对依赖进行统一管理

聚合

Maven 聚合是指将多个项目组织到一个父级项目中，以便一起构建和管理的机制。聚合可以帮助我们更好地管理一组相关的子项目，同时简化它们的构建和部署过程。

  • 聚合作用
    1. 管理多个子项目：通过聚合，可以将多个子项目组织在一起，方便管理和维护。
    2. 构建和发布一组相关的项目：通过聚合，可以在一个命令中构建和发布多个相关的项目，简化了部署和维护工作。
    3. 优化构建顺序：通过聚合，可以对多个项目进行顺序控制，避免出现构建依赖混乱导致构建失败的情况。
    4. 统一管理依赖项：通过聚合，可以在父项目中管理公共依赖项和插件，避免重复定义。
